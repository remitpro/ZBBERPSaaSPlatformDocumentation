    Technical Architecture Document¶

    ZBB ERP Module - SaaS Platform¶

    Version: 1.0
    Date: February 2026
    Status: Production Architecture

    Executive Summary¶

    This document describes the technical architecture for the Zero-Based Budgeting (ZBB) ERP Module delivered as a multi-tenant SaaS platform. The architecture is designed for scalability, security, high availability, and performance, supporting 1000+ concurrent users across multiple financial institutions.

    Architecture Overview¶

    High-Level Architecture¶

    
                        Client Layer                              

  Web App (React)    Mobile App (iOS/Android)    API Clients   

                               HTTPS/WSS

                     CDN & Load Balancer                          
              (CloudFlare / AWS CloudFront)                       

                              

                    API Gateway Layer                             

  Authentication    Rate Limiting    Request Routing    SSL    

                              

                   Application Layer (Microservices)              

  Budget Service    Loan Service    Funding Service    etc.   

                              

                      Data Layer                                  

  PostgreSQL (Primary)    Redis Cache    S3 Object Storage

    1. Client Layer¶

    1.1 Web Application¶

    Technology Stack: - Framework: React 18.x with TypeScript - State Management: Redux Toolkit + RTK Query - UI Library: Material-UI (MUI) v5 - Charts: Recharts + D3.js - Forms: React Hook Form + Yup validation - Build Tool: Vite - Testing: Jest + React Testing Library

    Key Features: - Responsive design (desktop, tablet) - Progressive Web App (PWA) capabilities - Offline mode with service workers - Real-time updates via WebSockets - Code splitting and lazy loading

    Performance Targets: - First Contentful Paint (FCP): <1.5s - Time to Interactive (TTI): <3.5s - Lighthouse Score: >90

    1.2 Mobile Application¶

    Technology Stack: - Framework: React Native 0.72+ - Navigation: React Navigation v6 - State Management: Redux Toolkit - UI Components: React Native Paper - Offline Storage: AsyncStorage + WatermelonDB

    Platforms: - iOS 14+ (Swift bridge for native features) - Android 10+ (Kotlin bridge for native features)

    Key Features: - Biometric authentication (Face ID, Touch ID, Fingerprint) - Push notifications (Firebase Cloud Messaging) - Offline-first architecture with sync - Camera integration for document upload - Native performance optimization

    1.3 API Clients¶

    Supported Integrations: - REST API (OpenAPI 3.0 specification) - GraphQL API (Apollo Server) - WebSocket API (Socket.io) - Webhook endpoints for event notifications

    2. Infrastructure Layer¶

    2.1 Cloud Provider: AWS (Primary)¶

    Region Strategy: - Primary: US-East-1 (N. Virginia) - Secondary: EU-West-1 (Ireland) for EU customers - DR Site: US-West-2 (Oregon)

    Services Used: - Compute: EKS (Kubernetes) for container orchestration - Database: RDS PostgreSQL Multi-AZ - Cache: ElastiCache Redis Cluster - Storage: S3 for object storage - CDN: CloudFront - DNS: Route 53 - Monitoring: CloudWatch + Prometheus + Grafana - Secrets: AWS Secrets Manager - Queue: SQS + SNS for messaging

    2.2 Container Orchestration¶

    Kubernetes (EKS): - Cluster Size: 3 master nodes, 10-50 worker nodes (auto-scaling) - Node Types: - Compute-optimized (c5.2xlarge) for API services - Memory-optimized (r5.xlarge) for data processing - GPU instances (p3.2xlarge) for AI/ML workloads

    Deployment Strategy: - Blue-Green deployments for zero downtime - Canary releases for gradual rollout - Horizontal Pod Autoscaler (HPA) based on CPU/memory - Cluster Autoscaler for node scaling

    2.3 Load Balancing & CDN¶

    Application Load Balancer (ALB): - SSL/TLS termination - Path-based routing - Health checks - Sticky sessions for WebSocket connections

    CloudFront CDN: - Global edge locations (200+ PoPs) - Static asset caching (JS, CSS, images) - Gzip/Brotli compression - DDoS protection via AWS Shield

    3. Application Layer (Microservices)¶

    3.1 Microservices Architecture¶

    Core Services:

    3.1.1 Authentication Service¶

    • Technology: Node.js + Express + Passport.js
    • Responsibilities:
    • User authentication (username/password, SSO, SAML 2.0)
    • Multi-Factor Authentication (MFA) via TOTP
    • JWT token generation and validation
    • Session management
    • OAuth 2.0 provider for third-party integrations
    • Database: PostgreSQL (users, roles, permissions)
    • Cache: Redis (sessions, tokens)

    3.1.2 Budget Service¶

    • Technology: .NET 8 (C#) + ASP.NET Core
    • Responsibilities:
    • Budget creation and management
    • Decision package grouping
    • Budget templates and validation
    • Budget consolidation and rollup
    • Variance analysis
    • Database: PostgreSQL (budget data)
    • Cache: Redis (frequently accessed budgets)
    • Message Queue: SQS (async processing)

    3.1.3 Loan Portfolio Service¶

    • Technology: Java 17 + Spring Boot 3.x
    • Responsibilities:
    • Loan product management (15 products)
    • IFRS 9 provisioning calculations
    • Monthly phasing calculations
    • Loan repricing and early repayment logic
    • Interest income calculations
    • Database: PostgreSQL (loan data)
    • Cache: Redis (calculation results)
    • Performance: Batch processing for large portfolios

    3.1.4 Funding Service¶

    • Technology: Python 3.11 + FastAPI
    • Responsibilities:
    • Deposit and borrowing management
    • Maturity bucket analysis
    • Liquidity ratio calculations (LCR, NSFR, LDR)
    • Interest expense calculations
    • Debt maturity profile
    • Database: PostgreSQL (funding data)
    • Cache: Redis (ratio calculations)

    3.1.5 Financial Statements Service¶

    • Technology: .NET 8 (C#) + ASP.NET Core
    • Responsibilities:
    • Income Statement generation
    • Balance Sheet generation
    • Cash Flow Statement (indirect method)
    • Automated consolidation
    • Financial statement validation
    • Database: PostgreSQL (financial data)
    • Cache: Redis (consolidated statements)

    3.1.6 KPI Service¶

    • Technology: Python 3.11 + FastAPI
    • Responsibilities:
    • KPI calculations (16 KPIs)
    • Target and threshold management
    • Alert generation
    • Trend analysis
    • KPI dashboard data aggregation
    • Database: PostgreSQL (KPI data)
    • Cache: Redis (real-time KPI values)
    • Time-Series DB: InfluxDB (historical KPI trends)

    3.1.7 Reporting Service¶

    • Technology: Node.js + Express
    • Responsibilities:
    • Standard report generation (25 reports)
    • Custom report builder
    • Excel export (ExcelJS library)
    • PDF export (Puppeteer for rendering)
    • Report scheduling and distribution
    • Database: PostgreSQL (report definitions)
    • Storage: S3 (generated reports)
    • Queue: SQS (async report generation)

    3.1.8 Scenario Planning Service¶

    • Technology: Python 3.11 + FastAPI + NumPy/SciPy
    • Responsibilities:
    • What-if analysis engine
    • Sensitivity analysis
    • Monte Carlo simulation (1000+ iterations)
    • Scenario comparison
    • Predictive analytics
    • Database: PostgreSQL (scenarios)
    • Compute: GPU instances for complex simulations
    • Cache: Redis (simulation results)

    3.1.9 Workflow Service¶

    • Technology: Node.js + Express + Bull (job queue)
    • Responsibilities:
    • 8-stage approval workflow
    • SLA monitoring and alerts
    • Email notifications (SendGrid)
    • Workflow state management
    • Escalation logic
    • Database: PostgreSQL (workflow states)
    • Queue: Redis (Bull queues)
    • Email: SendGrid API

    3.1.10 GL Integration Service¶

    • Technology: Java 17 + Spring Boot + Apache Camel
    • Responsibilities:
    • GL data synchronization (CoA, master data)
    • Budget upload to GL
    • Actual data retrieval
    • Error handling and retry logic
    • Integration monitoring
    • Database: PostgreSQL (integration logs)
    • Queue: SQS (async integration jobs)
    • Cache: Redis (GL data cache)

    3.1.11 AI/ML Service¶

    • Technology: Python 3.11 + FastAPI + TensorFlow/PyTorch
    • Responsibilities:
    • Budget recommendations (ML models)
    • Anomaly detection (Isolation Forest)
    • Natural language query (NLP with BERT)
    • Predictive analytics (time series forecasting)
    • Smart data validation
    • Database: PostgreSQL (training data)
    • Model Storage: S3 (trained models)
    • Compute: GPU instances (p3.2xlarge)

    3.1.12 Audit Service¶

    • Technology: Go + Gin framework
    • Responsibilities:
    • Comprehensive audit logging (20 action types)
    • Audit trail queries
    • Compliance reporting
    • Data retention management
    • Audit log export
    • Database: PostgreSQL (audit logs)
    • Time-Series DB: InfluxDB (high-volume logs)
    • Storage: S3 (archived logs)

    3.2 Service Communication¶

    Synchronous Communication: - REST API (HTTP/HTTPS) - gRPC for internal service-to-service calls (high performance)

    Asynchronous Communication: - Message Queue (SQS) for long-running tasks - Event Bus (SNS) for event-driven architecture - WebSockets (Socket.io) for real-time updates

    Service Mesh: - Istio for service discovery, load balancing, and observability - mTLS for secure service-to-service communication

    4. Data Layer¶

    4.1 Primary Database: PostgreSQL 14+¶

    Configuration: - Multi-AZ deployment for high availability - Read replicas (3) for read-heavy workloads - Connection pooling (PgBouncer) - Automated backups (daily, 30-day retention) - Point-in-time recovery (PITR)

    Database Schema: - Multi-Tenancy: Tenant isolation via tenant_id column + Row-Level Security (RLS) - Partitioning: Table partitioning by tenant and date for large tables - Indexing: B-tree, GiST, and partial indexes for performance

    Key Tables: - tenants - Tenant configuration - users - User accounts - budget_header - Budget master data - loan_budget - Loan portfolio budgets - funding_budget - Deposit and borrowing budgets - financial_statements - Generated statements - kpis - KPI values and history - audit_logs - Comprehensive audit trail

    Performance Optimization: - Query optimization with EXPLAIN ANALYZE - Materialized views for complex aggregations - Caching frequently accessed data in Redis

    4.2 Cache Layer: Redis 7+¶

    Configuration: - Redis Cluster (6 nodes: 3 masters, 3 replicas) - Persistence: RDB snapshots + AOF (Append-Only File) - Eviction policy: LRU (Least Recently Used)

    Use Cases: - Session storage (TTL: 24 hours) - JWT token blacklist - Frequently accessed data (budgets, KPIs) - Calculation result caching - Rate limiting counters - Real-time collaboration state

    Cache Invalidation: - Event-driven invalidation on data updates - TTL-based expiration - Manual invalidation via admin API

    4.3 Object Storage: AWS S3¶

    Buckets: - zbb-documents - User-uploaded documents (invoices, attachments) - zbb-reports - Generated reports (Excel, PDF) - zbb-backups - Database backups - zbb-ml-models - Trained ML models - zbb-audit-archives - Archived audit logs

    Configuration: - Versioning enabled - Lifecycle policies (transition to Glacier after 90 days) - Server-side encryption (SSE-S3) - Cross-region replication for DR

    4.4 Time-Series Database: InfluxDB¶

    Use Cases: - KPI historical trends - System metrics (CPU, memory, response times) - High-volume audit logs - Performance monitoring data

    Retention Policies: - Raw data: 90 days - Aggregated data (hourly): 1 year - Aggregated data (daily): 5 years

    5. Security Architecture¶

    5.1 Authentication & Authorization¶

    Authentication Methods: - Username/Password (bcrypt hashing) - Single Sign-On (SSO) via SAML 2.0 - OAuth 2.0 (Google, Microsoft, Okta) - Multi-Factor Authentication (TOTP via Google Authenticator) - Biometric (mobile app)

    Authorization: - Role-Based Access Control (RBAC) - 6 roles - Attribute-Based Access Control (ABAC) for fine-grained permissions - Row-Level Security (RLS) in PostgreSQL for tenant isolation

    Token Management: - JWT tokens (access token: 15 min, refresh token: 7 days) - Token rotation on refresh - Token blacklist in Redis for logout

    5.2 Data Security¶

    Encryption: - At Rest: AES-256 encryption for databases and S3 - In Transit: TLS 1.3 for all communications - Application-Level: Field-level encryption for sensitive data (PII)

    Data Masking: - Sensitive data masked in logs - PII redaction in non-production environments

    Compliance: - GDPR compliance (data privacy, right to erasure) - SOC 2 Type II certification - ISO 27001 compliance - PCI DSS for payment data (if applicable)

    5.3 Network Security¶

    Firewall: - AWS Security Groups (stateful firewall) - Network ACLs for subnet-level control - WAF (Web Application Firewall) for OWASP Top 10 protection

    DDoS Protection: - AWS Shield Standard (automatic) - AWS Shield Advanced (optional for enterprise customers) - Rate limiting at API Gateway (1000 req/min per user)

    VPN & Private Connectivity: - AWS PrivateLink for customer VPC connections - VPN for secure admin access - Bastion hosts for SSH access (no direct internet access)

    5.4 Vulnerability Management¶

    Security Scanning: - Container image scanning (Trivy, Snyk) - Dependency scanning (OWASP Dependency-Check) - SAST (Static Application Security Testing) - SonarQube - DAST (Dynamic Application Security Testing) - OWASP ZAP - Penetration testing (annual, by third-party)

    Patch Management: - Automated OS patching (AWS Systems Manager) - Monthly security updates for dependencies - Zero-day vulnerability response plan

    6. Scalability & Performance¶

    6.1 Horizontal Scaling¶

    Auto-Scaling Policies: - Application Layer: Scale out when CPU >70% or memory >80% - Database: Read replicas scale based on read load - Cache: Redis cluster scales based on memory usage

    Load Distribution: - Round-robin load balancing - Least connections for WebSocket - Geographic routing for global customers

    6.2 Performance Optimization¶

    Caching Strategy: - L1 Cache: In-memory application cache (5 min TTL) - L2 Cache: Redis distributed cache (1 hour TTL) - L3 Cache: CDN edge cache (24 hour TTL for static assets)

    Database Optimization: - Connection pooling (max 100 connections per service) - Query optimization and indexing - Materialized views for complex reports - Partitioning for large tables (>10M rows)

    Async Processing: - Long-running tasks (reports, simulations) processed asynchronously - Job queues with priority levels - Progress tracking and notifications

    6.3 Performance Targets¶

    7. High Availability & Disaster Recovery¶

    7.1 High Availability¶

    Application Layer: - Multi-AZ deployment (3 availability zones) - Minimum 3 replicas per service - Health checks and automatic failover - Circuit breakers for fault tolerance

    Database Layer: - Multi-AZ RDS with automatic failover (RPO: 0, RTO: <2 min) - Read replicas in multiple AZs - Automated backups every 6 hours

    Cache Layer: - Redis Cluster with replication - Automatic failover to replica nodes

    7.2 Disaster Recovery¶

    Backup Strategy: - Database: Daily full backups + continuous WAL archiving - Object Storage: Cross-region replication to DR site - Configuration: Infrastructure as Code (Terraform) in Git

    Recovery Objectives: - RPO (Recovery Point Objective): <1 hour - RTO (Recovery Time Objective): <4 hours

    DR Procedures: 1. Automated failover to DR region (US-West-2) 2. DNS update via Route 53 health checks 3. Database restore from latest backup 4. Application deployment from container registry 5. Validation and smoke testing

    DR Testing: - Quarterly DR drills - Annual full failover test

    8. Monitoring & Observability¶

    8.1 Monitoring Stack¶

    Metrics: - Prometheus: Metrics collection (CPU, memory, request rates) - Grafana: Visualization and dashboards - CloudWatch: AWS service metrics

    Logging: - ELK Stack: Elasticsearch + Logstash + Kibana - Fluentd: Log aggregation from containers - CloudWatch Logs: AWS service logs

    Tracing: - Jaeger: Distributed tracing - OpenTelemetry: Instrumentation

    Alerting: - PagerDuty: On-call alerting - Slack: Team notifications - Email: Critical alerts

    8.2 Key Metrics Monitored¶

    Application Metrics: - Request rate, error rate, latency (RED metrics) - Active users, session duration - API endpoint performance - Service-to-service communication latency

    Infrastructure Metrics: - CPU, memory, disk usage - Network throughput - Pod/container health - Database connections and query performance

    Business Metrics: - User adoption rate - Feature usage statistics - Budget creation rate - Report generation time

    8.3 Alerting Rules¶

    Critical Alerts (PagerDuty): - System availability <99% - Database connection failures - API error rate >5% - Security incidents

    Warning Alerts (Slack): - CPU usage >80% - Memory usage >85% - Slow queries (>1s) - High cache miss rate (>20%)

    9. DevOps & CI/CD¶

    9.1 CI/CD Pipeline¶

    Source Control: - Git: GitHub Enterprise - Branching Strategy: GitFlow (main, develop, feature, hotfix) - Code Review: Pull requests with 2 approvals required

    CI Pipeline (GitHub Actions): 1. Code checkout 2. Dependency installation 3. Linting (ESLint, Pylint, SonarQube) 4. Unit tests (Jest, pytest, JUnit) 5. Integration tests 6. Security scanning (Snyk, Trivy) 7. Build Docker images 8. Push to container registry (ECR)

    CD Pipeline: 1. Deploy to Dev environment (automatic) 2. Automated smoke tests 3. Deploy to Staging (automatic) 4. Automated E2E tests (Cypress, Selenium) 5. Deploy to Production (manual approval) 6. Blue-Green deployment 7. Health checks and rollback if needed

    9.2 Infrastructure as Code¶

    Tools: - Terraform: Infrastructure provisioning (AWS resources) - Helm: Kubernetes application deployment - Ansible: Configuration management

    Environments: - Development: Shared environment for developers - Staging: Production-like environment for testing - Production: Live customer environment

    9.3 Release Strategy¶

    Release Cadence: - Major Releases: Quarterly (new features) - Minor Releases: Monthly (enhancements, bug fixes) - Patch Releases: Weekly (critical bug fixes) - Hotfixes: As needed (security vulnerabilities)

    Deployment Windows: - Weekdays: 2 AM - 4 AM UTC (low traffic period) - Weekends: Avoided unless critical

    10. Multi-Tenancy¶

    10.1 Tenant Isolation¶

    Database Isolation: - Shared database with tenant_id column - Row-Level Security (RLS) enforced at database level - Separate schemas per tenant (optional for enterprise customers)

    Application Isolation: - Tenant context in JWT token - Middleware validates tenant access - Cross-tenant data access prevented

    Resource Isolation: - Kubernetes namespaces per tenant (enterprise tier) - Resource quotas and limits - Network policies for isolation

    10.2 Tenant Onboarding¶

    Provisioning Process: 1. Tenant registration via admin portal 2. Database schema initialization 3. Default configuration setup 4. Admin user creation 5. Welcome email with login credentials 6. Onboarding wizard for initial setup

    Provisioning Time: <5 minutes (automated)

    11. API Architecture¶

    11.1 REST API¶

    Design Principles: - RESTful conventions (GET, POST, PUT, DELETE) - Resource-based URLs (/api/v1/budgets/{id}) - JSON request/response format - HATEOAS for discoverability

    Versioning: - URL-based versioning (/api/v1/, /api/v2/) - Backward compatibility for 2 major versions

    Rate Limiting: - 1000 requests/minute per user - 10,000 requests/minute per tenant - 429 Too Many Requests response

    API Documentation: - OpenAPI 3.0 specification - Swagger UI for interactive documentation - Postman collection for testing

    11.2 GraphQL API¶

    Schema: - Type-safe schema with GraphQL SDL - Queries for data retrieval - Mutations for data modification - Subscriptions for real-time updates

    Performance: - DataLoader for batching and caching - Query complexity analysis - Depth limiting (max 10 levels)

    11.3 WebSocket API¶

    Use Cases: - Real-time collaboration (multiple users editing) - Live KPI updates - Workflow notifications - Chat and comments

    Technology: - Socket.io for WebSocket management - Redis pub/sub for message distribution - Sticky sessions for connection persistence

    12. Mobile Architecture¶

    12.1 React Native App¶

    Architecture Pattern: - Redux for state management - Redux Saga for side effects - React Navigation for routing

    Offline Support: - WatermelonDB for local database - Sync queue for offline changes - Conflict resolution on sync

    Performance: - Code splitting and lazy loading - Image optimization and caching - Native module bridges for performance-critical code

    12.2 Push Notifications¶

    Provider: - Firebase Cloud Messaging (FCM) for Android - Apple Push Notification Service (APNs) for iOS

    Notification Types: - Workflow approvals - Budget deadline reminders - KPI threshold alerts - System announcements

    13. AI/ML Architecture¶

    13.1 ML Pipeline¶

    Training Pipeline: 1. Data extraction from PostgreSQL 2. Feature engineering (Pandas, NumPy) 3. Model training (TensorFlow, scikit-learn) 4. Model evaluation and validation 5. Model versioning (MLflow) 6. Model deployment to S3

    Inference Pipeline: 1. Real-time prediction via REST API 2. Batch prediction via scheduled jobs 3. Model monitoring and drift detection 4. A/B testing for model comparison

    13.2 ML Models¶

    Budget Recommendation Model: - Algorithm: Gradient Boosting (XGBoost) - Features: Historical budgets, actuals, trends - Accuracy: 85%

    Anomaly Detection Model: - Algorithm: Isolation Forest - Features: Budget amounts, variance, patterns - Precision: 92%, Recall: 88%

    NLP Model (Natural Language Query): - Algorithm: BERT fine-tuned - Use Case: "Show me budgets over $1M" - Accuracy: 78%

    14. Cost Optimization¶

    14.1 Cost Management¶

    Compute: - Spot instances for non-critical workloads (70% cost savings) - Reserved instances for baseline capacity (40% cost savings) - Auto-scaling to match demand

    Storage: - S3 lifecycle policies (Glacier for archives) - Database storage optimization (compression, partitioning) - CDN caching to reduce origin requests

    Monitoring: - AWS Cost Explorer for cost analysis - Budget alerts for overspending - Tagging for cost allocation by tenant

    15. Compliance & Governance¶

    15.1 Compliance Standards¶

    • GDPR: Data privacy, right to erasure, data portability
    • SOC 2 Type II: Security, availability, confidentiality
    • ISO 27001: Information security management
    • HIPAA: Healthcare data protection (if applicable)
