    Functional Requirements Document (FRD) and System Design Specification¶

    Zero-Based Budgeting ERP Module for Banking Institution¶

    Document Version: 1.0
    Date: February 10, 2026
    Classification: Procurement - Request for Proposal (RFP)
    Prepared For: ERP Vendor Selection and Implementation

    Executive Summary¶

    This Functional Requirements Document (FRD) defines the comprehensive requirements for implementing an automated Zero-Based Budgeting (ZBB) module within an Enterprise Resource Planning (ERP) system for a banking institution. The solution must replace the current manual, Excel-based budget process with an integrated, automated system that supports budget formulation, coordination, consolidation, and automated generation of complete financial statements and banking-specific Key Performance Indicators (KPIs).

    Key Objectives¶

    • Automate the end-to-end budget process from assumptions setup to financial statement generation
    • Enforce Zero-Based Budgeting principles with mandatory justifications
    • Integrate seamlessly with the General Ledger and Chart of Accounts
    • Generate the complete Three Statement Model (Income Statement, Balance Sheet, Cash Flow Statement)
    • Calculate banking-specific KPIs automatically with full traceability
    • Provide robust workflow, approval mechanisms, and comprehensive audit trails

    1. Current State Analysis¶

    1.1 Existing Budget Framework¶

    The Bank currently employs a Zero-Based Budgeting (ZBB) methodology for annual budget preparation, requiring all business units to justify anticipated expenditures from a zero base. The process is governed by centrally approved budget assumptions including inflation rates, growth targets, and regulatory constraints.

    1.2 Current Process Flow¶

    flowchart TD
    A[Budget Team: Prepare Excel Templates] --> B[Distribute to Business Units]
    B --> C[Heads of Units: Populate Templates]
    B --> D[Operations Team: Loan Portfolio Budget]
    C --> E[Submit to Budget Team]
    D --> E
    E --> F[Manual Consolidation in Master Template]
    F --> G[Generate Draft Summaries]
    G --> H[Manual Financial Statement Preparation]
    H --> I[Manual Ratio Calculations]

    1.3 Key Pain Points¶

    Challenge Impact
    Heavy reliance on manual Excel templates High error risk, version control issues
    Manual consolidation process Time-consuming, labor-intensive
    Limited audit trail Compliance and governance risks
    No automated approval workflow Delays, lack of accountability
    Manual financial statement preparation Inconsistencies, delayed reporting
    Difficulty in scenario analysis Limited strategic planning capability

    2. System Integration & Architecture¶

    2.1 Integration with General Ledger (GL)¶

    2.1.1 Chart of Accounts Mapping¶

    Requirement: The Budget Module must be fully integrated with the Bank's General Ledger through a synchronized Chart of Accounts (CoA) mapping.

    Specifications:

    • Bi-directional synchronization between Budget Module and GL
    • All budget line items must map to specific GL account codes
    • Support for multi-dimensional account structures:
    • Cost Center
    • Business Unit
    • Product Line
    • Geographic Location
    • Project/Initiative Code
    • Real-time validation to ensure budget entries align with active GL accounts
    • Ability to handle GL account hierarchy (parent-child relationships)
    • Support for account consolidation rules at multiple organizational levels

    2.1.2 Data Flow Architecture¶

    flowchart LR
    A[Budget Module] <-->|CoA Sync| B[General Ledger]
    A <-->|Actual vs Budget| C[Financial Reporting]
    A -->|Budget Data| D[Data Warehouse]
    B -->|Actual Data| D
    D -->|Analytics| E[BI Dashboard]

    Integration Points:

    1. Master Data Synchronization
    2. Chart of Accounts
    3. Cost Centers
    4. Business Units

    5. Employee Master (for approvers)

    6. Transactional Data Flow

    7. Approved budget data flows to GL as budget allocations

    8. Actual expenditure data flows from GL for variance analysis

    9. Reporting Integration

    10. Consolidated financial statements pull from both budget and actual data
    11. KPI calculations use integrated data sources

    2.1.3 Technical Integration Requirements¶

    • API-based integration with RESTful web services
    • Real-time or near-real-time data synchronization (max 15-minute latency)
    • Error handling and reconciliation mechanisms
    • Data integrity checks at integration points
    • Support for batch and real-time processing modes

    2.2 Role-Based Access Control (RBAC) Model¶

    2.2.1 User Roles and Permissions Matrix¶

    Role Access Level Permissions
    Executive Management View, Approve (Final) - View all budgets across the Bank
    - Approve/reject consolidated budgets
    - Access all reports and dashboards
    - Configure strategic assumptions
    Budget Team Admin, Review, Consolidate - Create and distribute budget templates
    - Set up budget assumptions
    - Review all unit submissions
    - Query and request revisions
    - Consolidate budgets
    - Generate financial statements
    - Configure validation rules
    Heads of Business Units Create, Edit, Submit - Access assigned unit templates
    - Enter budget data for their units
    - Attach justification documents
    - Submit for review
    - Respond to queries
    - View their unit's budget status
    Operations Team Create, Edit, Submit (Loan Portfolio) - Access loan portfolio budget templates
    - Enter disbursement projections by PFI and product
    - Input interest income assumptions
    - Submit loan budget
    - View loan portfolio reports
    Finance Team View, Analyze - View approved budgets
    - Run variance analysis
    - Generate ad-hoc reports
    - Access financial statements
    Auditors View (Read-only), Audit Trail - Read-only access to all budget data
    - Full audit trail access
    - Export capabilities for audit purposes

    2.2.2 Security Requirements¶

    Authentication: - Integration with Bank's existing Active Directory/LDAP - Support for Single Sign-On (SSO) - Multi-factor authentication (MFA) for privileged users

    Authorization: - Granular permissions at field level - Data segregation by business unit - Hierarchical access control (managers can view subordinate budgets)

    Data Security: - Encryption at rest (AES-256) - Encryption in transit (TLS 1.3) - Field-level encryption for sensitive data - Compliance with banking data protection regulations

    2.2.3 Delegation and Proxy Access¶

    • Ability to delegate approval authority temporarily
    • Proxy access for users during absence
    • Audit trail for all delegated actions

    3. Functional Module Requirements¶

    3.1 Assumptions Engine¶

    3.1.1 Central Assumptions Dashboard¶

    Purpose: Provide a centralized interface for the Budget Team to define, manage, and version-control all budget assumptions that drive calculations across the system.

    Required Assumptions Categories:

    1. Macroeconomic Assumptions
    2. Inflation rate (general and sector-specific)
    3. Interest rate benchmarks (policy rate, LIBOR, etc.)
    4. Exchange rates (for multi-currency operations)

    5. GDP growth projections

    6. Growth Targets

    7. Loan portfolio growth rate (by product type)
    8. Deposit growth rate
    9. Revenue growth targets

    10. Market share objectives

    11. Regulatory and Policy Constraints

    12. Capital adequacy requirements (CAR)
    13. Liquidity coverage ratio (LCR) requirements
    14. Maximum loan-to-deposit ratio
    15. Provisioning rates by loan classification

    16. Regulatory expense caps

    17. Operational Assumptions

    18. Headcount growth/reduction plans
    19. Salary increment percentages
    20. Branch expansion/closure plans
    21. Technology investment priorities

    3.1.2 Functional Requirements¶

    FR-ASM-001: The system shall provide a dedicated Assumptions Management module accessible only to authorized Budget Team members.

    FR-ASM-002: The system shall support creation of multiple assumption scenarios (Base Case, Optimistic, Pessimistic).

    FR-ASM-003: The system shall maintain complete version history of all assumptions with: - Version number - Effective date - Created by (user ID) - Timestamp - Change description - Approval status

    FR-ASM-004: The system shall allow assumptions to be defined at multiple organizational levels: - Bank-wide (default) - Business unit-specific - Product-specific - Geographic region-specific

    FR-ASM-005: The system shall enforce approval workflow for assumption changes: - Draft Review Approved Published - Only published assumptions are available for budget preparation

    FR-ASM-006: The system shall automatically cascade assumption changes to all dependent budget calculations with user notification.

    FR-ASM-007: The system shall provide comparison reports showing impact of assumption changes on budget outcomes.

    3.1.3 User Interface Requirements¶

    • Dashboard view showing all active assumptions by category
    • Drill-down capability to view assumption details and history
    • Visual indicators for assumptions pending approval
    • Impact analysis tool showing which budgets are affected by each assumption
    • Export functionality for assumption documentation

    3.2 Zero-Based Budgeting (ZBB) Logic Module¶

    3.2.1 ZBB Principles Enforcement¶

    Purpose: Ensure that every expense line item is justified from a zero base, eliminating automatic rollover of prior year budgets.

    FR-ZBB-001: The system shall NOT auto-populate budget templates with prior year actual or budget figures as default values.

    FR-ZBB-002: The system shall provide prior year data as reference only, displayed in a separate read-only column for comparison purposes.

    FR-ZBB-003: The system shall require explicit justification for every non-zero budget line item.

    3.2.2 Digital Budget Templates¶

    FR-ZBB-004: The system shall provide role-based digital budget templates dynamically generated based on: - User role (Business Unit Head, Operations Team, etc.) - Organizational unit assignment - Chart of Accounts access permissions

    FR-ZBB-005: Templates shall include the following columns for Operating Expenses:

    Column Type Required Description
    GL Account Code Auto-populated Yes From CoA mapping
    Account Description Auto-populated Yes From CoA
    Budget Amount User input Yes Proposed budget for the line item
    Justification Text field (min 50 chars) Yes Mandatory explanation
    Business Driver Dropdown Yes Volume, Headcount, Strategic Initiative, Regulatory, etc.
    Calculation Basis Text/Formula Conditional Required if amount > threshold
    Supporting Documents File attachment Conditional Required for amounts exceeding defined threshold
    Prior Year Actual Display only No For reference
    Prior Year Budget Display only No For reference

    FR-ZBB-006: The system shall enforce mandatory justification fields with the following rules: - Minimum character count (configurable, default 50 characters) - Cannot submit with placeholder text (e.g., "TBD", "N/A") - Must select a valid business driver from predefined list

    FR-ZBB-007: The system shall support attachment of multiple supporting documents per line item: - Accepted formats: PDF, Excel, Word, images - Maximum file size: 10MB per attachment - Virus scanning on upload - Version control for attachments

    3.2.3 Decision Package Approach¶

    FR-ZBB-008: The system shall support grouping of related line items into "Decision Packages" representing discrete activities or initiatives.

    FR-ZBB-009: Each Decision Package shall have: - Package name and description - Package owner - Total cost - Consolidated justification - Priority ranking (Critical, High, Medium, Low) - Incremental levels (Minimum, Base, Enhanced)

    FR-ZBB-010: The system shall allow Budget Team to approve/reject entire Decision Packages or individual line items within packages.

    3.3 Operations/Loan Portfolio Module¶

    3.3.1 Purpose¶

    Enable the Operations Team to prepare comprehensive loan portfolio budgets including disbursement projections, portfolio growth, and interest income calculations.

    3.3.2 Disbursement Projections¶

    FR-LOAN-001: The system shall provide a Loan Disbursement Budget template with the following dimensions:

    Dimension 1: Product Type - Retail Loans (Personal, Auto, Mortgage, etc.) - SME Loans - Corporate Loans - Agricultural Loans - Other specialized products

    Dimension 2: Participating Financial Institution (PFI) Type - Tier 1 Banks - Tier 2 Banks - Microfinance Institutions - Non-Bank Financial Institutions - Direct Lending (if applicable)

    Dimension 3: Individual PFI Names - Dropdown list of all active PFIs - Ability to add new PFIs

    FR-LOAN-002: For each combination of Product Type PFI Type PFI Name, the system shall capture:

    Field Type Required Description
    Projected Disbursement Amount Currency Yes Total loan amount to be disbursed
    Number of Loans Integer Yes Expected loan count
    Average Loan Size Auto-calculated N/A Disbursement Amount Number of Loans
    Disbursement Timing Monthly distribution Yes Phasing across 12 months
    Interest Rate Percentage Yes Expected lending rate
    Loan Tenor Months Yes Average loan duration
    Repayment Schedule Dropdown Yes Monthly, Quarterly, Bullet, etc.
    Expected NPL Rate Percentage Yes Projected non-performing loan rate
    Provisioning Rate Auto-calculated N/A Based on regulatory rules and NPL rate

    FR-LOAN-003: The system shall support monthly phasing of disbursements with: - Manual input option (enter amount per month) - Pattern-based distribution (equal, seasonal, custom curve) - Visual representation (chart showing disbursement profile)

    3.3.3 Loan Portfolio Growth Calculations¶

    FR-LOAN-004: The system shall automatically calculate projected loan portfolio growth:

    $$\text{Projected Loan Growth Rate} = \frac{\text{Total Projected Disbursements} - \text{Projected Repayments}}{\text{Opening Loan Portfolio Balance}} \times 100$$

    FR-LOAN-005: The system shall calculate projected loan portfolio balance by month:

    $$\text{Closing Balance}{\text{Month}} = \text{Opening Balance}{\text{Month}} + \text{Disbursements}{\text{Month}} - \text{Repayments}{\text{Month}} - \text{Write-offs}_{\text{Month}}$$

    FR-LOAN-006: Repayment projections shall be auto-calculated based on: - Existing portfolio repayment schedule (from loan management system) - New disbursements with specified tenor and repayment frequency - Historical prepayment rates (configurable assumption)

    3.3.4 Interest Income Calculations¶

    FR-LOAN-007: The system shall calculate projected interest income using the following logic:

    For each loan product/PFI combination:

    $$\text{Monthly Interest Income} = \text{Average Loan Balance}_{\text{Month}} \times \frac{\text{Interest Rate}}{12}$$

    Where:

    $$\text{Average Loan Balance}{\text{Month}} = \frac{\text{Opening Balance}{\text{Month}} + \text{Closing Balance}_{\text{Month}}}{2}$$

    FR-LOAN-008: Total Interest Income shall be aggregated across all products and PFIs and flow automatically to the Income Statement.

    FR-LOAN-009: The system shall calculate interest income on both accrual and cash basis, with clear labeling.

    3.3.5 Provisioning Calculations¶

    FR-LOAN-010: The system shall automatically calculate loan loss provisions based on: - Projected NPL rates (input by Operations Team) - Regulatory provisioning rates by loan classification (from Assumptions Engine) - Expected Credit Loss (ECL) methodology (IFRS 9 compliant)

    FR-LOAN-011: Provisioning calculation formula:

    $$\text{Provision Expense} = (\text{Projected NPL Amount} \times \text{Provisioning Rate}) - \text{Opening Provision Balance}$$

    FR-LOAN-012: Provision expense shall flow automatically to the Income Statement as operating expense.

    3.4 Borrowings and Funding Module¶

    3.4.1 Purpose¶

    Enable the Budget Team to prepare a comprehensive borrowings schedule reflecting anticipated sources of funding, pricing, timing, and associated interest expense.

    3.4.2 Funding Sources Template¶

    FR-FUND-001: The system shall provide a Borrowings Schedule template capturing:

    Field Type Required Description
    Funding Source Type Dropdown Yes Customer Deposits, Interbank Borrowing, Bond Issuance, Equity, Central Bank Facility, etc.
    Funding Source Name Text Yes Specific identifier (e.g., "5-Year Bond 2026")
    Principal Amount Currency Yes Total funding amount
    Drawdown Date Date Yes When funds are received
    Maturity Date Date Yes When principal is due
    Interest Rate Percentage Yes Cost of funds
    Interest Payment Frequency Dropdown Yes Monthly, Quarterly, Semi-Annual, Annual, At Maturity
    Fees and Charges Currency No Upfront or ongoing fees
    Collateral Required Yes/No Yes Whether secured

    FR-FUND-002: The system shall support multiple tranches for a single funding source with different terms.

    3.4.3 Interest Expense Calculations¶

    FR-FUND-003: The system shall automatically calculate interest expense for each funding source:

    $$\text{Monthly Interest Expense} = \text{Average Outstanding Balance}_{\text{Month}} \times \frac{\text{Interest Rate}}{12}$$

    FR-FUND-004: For customer deposits, the system shall support: - Different interest rates by deposit type (Savings, Current, Fixed Deposit) - Projected deposit balance growth (linked to deposit mobilization targets) - Weighted average deposit rate calculation

    FR-FUND-005: Total Interest Expense shall be automatically aggregated and flow to the Income Statement.

    3.4.4 Cost of Funds and Average Balance Calculations¶

    FR-FUND-006: The system shall calculate and display:

    $$\text{Weighted Average Cost of Funds} = \frac{\text{Total Interest Expense}}{\text{Average Total Liabilities}} \times 100$$

    $$\text{Average Total Liabilities}_{\text{Month}} = \frac{\text{Opening Liabilities} + \text{Closing Liabilities}}{2}$$

    FR-FUND-007: The system shall provide a monthly funding profile showing: - Opening balance by funding source - New drawdowns - Repayments - Interest accrued - Closing balance

    4. Automated Calculation & Integration Logic¶

    4.1 Three Statement Model Integration¶

    The ERP system must automatically generate a fully integrated Three Statement Model where changes in one statement dynamically flow through to the others, maintaining mathematical consistency.

    flowchart TD
    A[Budget Inputs] --> B[Income Statement]
    A --> C[Balance Sheet]
    B --> D[Net Income]
    D --> C
    D --> E[Cash Flow Statement]
    C --> E
    E --> F[Cash Balance]
    F --> C

    4.2 Income Statement Generation¶

    4.2.1 Income Statement Structure¶

    FR-IS-001: The system shall automatically generate a Budgeted Income Statement with the following structure:

    INCOME STATEMENT (Budgeted)
For the Year Ending [Budget Year]

INTEREST INCOME
  Interest on Loans                           [Auto-calculated from Loan Module]
  Interest on Investments                     [Input or auto-calculated]
  Other Interest Income                       [Input]
Total Interest Income                         [Sum]

INTEREST EXPENSE
  Interest on Customer Deposits               [Auto-calculated from Funding Module]
  Interest on Borrowings                      [Auto-calculated from Funding Module]
  Other Interest Expense                      [Input]
Total Interest Expense                        [Sum]

NET INTEREST INCOME                           [Total Interest Income - Total Interest Expense]

NON-INTEREST INCOME
  Fee and Commission Income                   [Input by Business Units]
  Trading Income                              [Input]
  Foreign Exchange Income                     [Input]
  Other Operating Income                      [Input]
Total Non-Interest Income                     [Sum]

TOTAL OPERATING INCOME                        [Net Interest Income + Non-Interest Income]

OPERATING EXPENSES
  Personnel Expenses                          [Consolidated from Business Unit budgets]
    - Salaries and Wages
    - Benefits and Allowances
    - Training and Development
  Administrative Expenses                     [Consolidated from Business Unit budgets]
    - Rent and Utilities
    - Technology and Communication
    - Marketing and Advertising
    - Professional Fees
    - Other Administrative Costs
  Depreciation and Amortization              [Auto-calculated from asset schedule]
  Loan Loss Provisions                        [Auto-calculated from Loan Module]
Total Operating Expenses                      [Sum]

PROFIT BEFORE TAX                             [Total Operating Income - Total Operating Expenses]

Income Tax Expense                            [Auto-calculated: PBT  Tax Rate]

PROFIT AFTER TAX                              [PBT - Tax Expense]

    4.2.2 Calculation Logic¶

    FR-IS-002: Interest Income calculation: - Sourced automatically from Loan Portfolio Module (FR-LOAN-007, FR-LOAN-008) - Investment interest calculated based on investment portfolio schedule - No manual override allowed; changes must be made at source

    FR-IS-003: Interest Expense calculation: - Sourced automatically from Borrowings and Funding Module (FR-FUND-003, FR-FUND-005) - No manual override allowed; changes must be made at source

    FR-IS-004: Operating Expenses consolidation: - Automatically aggregated from all approved Business Unit budgets - Mapped by GL account code - Drill-down capability to view source unit and justification

    FR-IS-005: Tax calculation: - Tax rate sourced from Assumptions Engine - Applied to Profit Before Tax - Support for deferred tax adjustments (input field)

    FR-IS-006: The system shall generate monthly, quarterly, and annual Income Statements.

    4.3 Balance Sheet Generation¶

    4.3.1 Balance Sheet Structure¶

    FR-BS-001: The system shall automatically generate a Budgeted Balance Sheet with the following structure:

    STATEMENT OF FINANCIAL POSITION (Budgeted)
As at [End of Budget Year]

ASSETS

    4.3.2 Calculation Logic and Dynamic Linkages¶

    FR-BS-002: Cash and Cash Equivalents: - Ending balance from Cash Flow Statement (FR-CF-001) - Automatically updated when cash flow changes

    FR-BS-003: Loans and Advances: - Closing balance from Loan Portfolio Module (FR-LOAN-005) - Calculated monthly and year-end balance displayed

    FR-BS-004: Loan Loss Provisions: - From Loan Module provisioning calculations (FR-LOAN-011) - Displayed as contra-asset

    FR-BS-005: Property, Plant and Equipment:

    $$\text{PPE}{\text{Closing}} = \text{PPE}{\text{Opening}} + \text{Capital Expenditure} - \text{Depreciation} - \text{Disposals}$$

    • Capital Expenditure from CAPEX budget (input by Business Units)
    • Depreciation auto-calculated based on depreciation schedule and rates

    FR-BS-006: Customer Deposits: - From Funding Module deposit projections - Closing balance at year-end

    FR-BS-007: Borrowings: - From Funding Module borrowings schedule - Sum of all outstanding borrowings at year-end

    FR-BS-008: Retained Earnings:

    $$\text{Retained Earnings}{\text{Closing}} = \text{Retained Earnings}{\text{Opening}} + \text{Profit After Tax} - \text{Dividends Declared}$$

    • Profit After Tax from Income Statement (FR-IS-001)
    • Dividends as input or based on dividend policy assumption

    FR-BS-009: The system shall enforce the fundamental accounting equation:

    $$\text{Total Assets} = \text{Total Liabilities} + \text{Total Equity}$$

    • Real-time validation with error flagging if imbalance exists
    • Drill-down to identify source of imbalance

    FR-BS-010: The system shall generate monthly Balance Sheet projections showing balance sheet evolution throughout the budget year.

    4.4.2 Calculation Logic¶

    FR-CF-002: Operating Activities - Adjustments: - Depreciation and Loan Loss Provisions added back (non-cash expenses) - Sourced automatically from Income Statement

    FR-CF-003: Operating Activities - Working Capital Changes:

    $$\Delta \text{Loans and Advances} = \text{Closing Net Loans} - \text{Opening Net Loans}$$

    • Increase in loans = cash outflow (negative)
    • Decrease in loans = cash inflow (positive)

    $$\Delta \text{Customer Deposits} = \text{Closing Deposits} - \text{Opening Deposits}$$

    • Increase in deposits = cash inflow (positive)
    • Decrease in deposits = cash outflow (negative)

    FR-CF-004: Investing Activities: - CAPEX sourced from Capital Expenditure budget - Investment purchases/sales from investment schedule

    FR-CF-005: Financing Activities: - New borrowings and repayments from Funding Module - Dividends from Retained Earnings calculation

    FR-CF-006: Ending Cash Balance: - Must reconcile with Cash and Cash Equivalents on Balance Sheet - System validation to ensure consistency

    FR-CF-007: The system shall generate monthly Cash Flow projections to identify potential liquidity gaps.

    4.5 Integration Validation Rules¶

    FR-INT-001: The system shall perform real-time validation checks:

    Validation Check Formula Action if Failed
    Balance Sheet Balance Total Assets = Total Liabilities + Equity Flag error, prevent finalization
    Cash Flow Reconciliation CF Ending Cash = BS Cash Balance Flag error, prevent finalization
    Net Income Flow IS Profit After Tax = Change in Retained Earnings (adjusted for dividends) Flag error, prevent finalization
    Interest Income Reconciliation Sum of Loan Module Interest = IS Interest Income Flag warning, allow override with justification

    FR-INT-002: The system shall provide a "Model Integrity Dashboard" showing: - Status of all integration checks (Pass/Fail) - Source of any discrepancies - Drill-down to specific line items causing issues

    5. Financial Reporting & KPI Engine¶

    5.1 Banking-Specific KPI Calculations¶

    The system must automatically calculate and display key banking performance metrics using standardized formulas. All KPIs must be traceable to source data with drill-down capability.

    5.1.1 Profitability Metrics¶

    FR-KPI-001: Net Interest Margin (NIM)

    $$\text{NIM} = \frac{\text{Interest Income} - \text{Interest Expense}}{\text{Average Earning Assets}} \times 100$$

    Where:

    $$\text{Average Earning Assets} = \frac{\text{Opening Earning Assets} + \text{Closing Earning Assets}}{2}$$

    Earning Assets = Loans + Investment Securities

    • Display: Percentage (2 decimal places)
    • Frequency: Monthly, Quarterly, Annual
    • Benchmark: Configurable target from Assumptions Engine
    • Drill-down: View components (Interest Income, Interest Expense, Earning Assets breakdown)

    FR-KPI-002: Return on Assets (ROA)

    $$\text{ROA} = \frac{\text{Net Profit After Tax}}{\text{Total Average Assets}} \times 100$$

    Where:

    $$\text{Total Average Assets} = \frac{\text{Opening Total Assets} + \text{Closing Total Assets}}{2}$$

    • Display: Percentage (2 decimal places)
    • Frequency: Quarterly, Annual (not meaningful monthly)
    • Benchmark: Configurable target
    • Drill-down: View Net Profit components and Asset composition

    FR-KPI-003: Return on Equity (ROE)

    $$\text{ROE} = \frac{\text{Net Profit After Tax}}{\text{Average Shareholders' Equity}} \times 100$$

    Where:

    $$\text{Average Shareholders' Equity} = \frac{\text{Opening Equity} + \text{Closing Equity}}{2}$$

    • Display: Percentage (2 decimal places)
    • Frequency: Quarterly, Annual
    • Benchmark: Configurable target
    • Drill-down: View Net Profit and Equity components

    FR-KPI-004: Cost-to-Income Ratio

    $$\text{Cost-to-Income Ratio} = \frac{\text{Operating Expenses}}{\text{Net Interest Income} + \text{Non-Interest Income}} \times 100$$

    Alternative formulation:

    $$\text{Cost-to-Income Ratio} = \frac{\text{Operating Expenses}}{\text{Total Operating Income}} \times 100$$

    • Display: Percentage (2 decimal places)
    • Frequency: Monthly, Quarterly, Annual
    • Benchmark: Configurable target (lower is better)
    • Drill-down: View Operating Expenses by category and Income components
    • Note: Exclude loan loss provisions from Operating Expenses for this calculation (configurable)

    5.1.2 Asset Quality Metrics¶

    FR-KPI-005: Loan Growth Rate

    $$\text{Loan Growth Rate} = \frac{\text{Closing Gross Loans} - \text{Opening Gross Loans}}{\text{Opening Gross Loans}} \times 100$$

    • Display: Percentage (2 decimal places)
    • Frequency: Monthly, Quarterly, Annual (annualized for monthly/quarterly)
    • Benchmark: Target from Assumptions Engine
    • Drill-down: View growth by product type, PFI, geographic region

    FR-KPI-006: Non-Performing Loan (NPL) Ratio

    $$\text{NPL Ratio} = \frac{\text{Gross Non-Performing Loans}}{\text{Gross Total Loans}} \times 100$$

    • Display: Percentage (2 decimal places)
    • Frequency: Monthly, Quarterly, Annual
    • Benchmark: Regulatory limit and internal target
    • Drill-down: View NPL by product type, vintage, PFI
    • Alert: Flag if projected NPL ratio exceeds regulatory or internal threshold

    FR-KPI-007: Provision Coverage Ratio

    $$\text{Provision Coverage Ratio} = \frac{\text{Total Loan Loss Provisions}}{\text{Gross Non-Performing Loans}} \times 100$$

    • Display: Percentage (2 decimal places)
    • Frequency: Monthly, Quarterly, Annual
    • Benchmark: Regulatory minimum (typically 100%+)
    • Drill-down: View provisions by loan classification
    • Alert: Flag if coverage falls below regulatory minimum

    5.1.3 Liquidity and Capital Metrics¶

    FR-KPI-008: Loan-to-Deposit Ratio (LDR)

    $$\text{LDR} = \frac{\text{Total Gross Loans}}{\text{Total Customer Deposits}} \times 100$$

    • Display: Percentage (2 decimal places)
    • Frequency: Monthly, Quarterly, Annual
    • Benchmark: Regulatory maximum and internal target
    • Drill-down: View loans and deposits composition
    • Alert: Flag if LDR exceeds regulatory limit

    FR-KPI-009: Liquidity Ratio

    $$\text{Liquidity Ratio} = \frac{\text{Liquid Assets}}{\text{Total Assets}} \times 100$$

    Where Liquid Assets = Cash + Cash Equivalents + Marketable Securities

    • Display: Percentage (2 decimal places)
    • Frequency: Monthly, Quarterly, Annual
    • Benchmark: Regulatory minimum and internal target
    • Drill-down: View liquid assets composition

    FR-KPI-010: Capital Adequacy Ratio (CAR) - Simplified

    $$\text{CAR} = \frac{\text{Total Equity}}{\text{Risk-Weighted Assets}} \times 100$$

    • Display: Percentage (2 decimal places)
    • Frequency: Quarterly, Annual
    • Benchmark: Regulatory minimum (e.g., 10% under Basel)
    • Note: Risk-weighted assets calculation requires input of risk weights by asset category
    • Alert: Flag if projected CAR falls below regulatory minimum

    5.1.4 Additional Performance Metrics¶

    FR-KPI-011: Earnings Per Share (EPS)

    $$\text{EPS} = \frac{\text{Net Profit After Tax}}{\text{Number of Outstanding Shares}}$$

    FR-KPI-012: Dividend Payout Ratio

    $$\text{Dividend Payout Ratio} = \frac{\text{Dividends Declared}}{\text{Net Profit After Tax}} \times 100$$

    FR-KPI-013: Net Interest Spread

    $$\text{Net Interest Spread} = \text{Average Lending Rate} - \text{Average Funding Rate}$$

    5.2 KPI Dashboard Requirements¶

    FR-KPI-014: The system shall provide an executive KPI dashboard displaying: - All key metrics in a single view - Visual indicators (green/yellow/red) based on performance vs. targets - Trend charts showing metric evolution over time - Comparison to prior year budget and actuals (when available)

    FR-KPI-015: The system shall allow users to: - Configure which KPIs appear on their personal dashboard - Set custom alert thresholds - Export KPI reports to Excel and PDF - Schedule automated KPI report distribution

    FR-KPI-016: All KPI calculations must be fully auditable with: - Formula displayed on hover/click - Drill-down to source data - Ability to export calculation details

    6. Workflow & Governance¶

    6.1 Budget Preparation and Approval Workflow¶

    6.1.1 Workflow Stages¶

    The system must support a structured, multi-stage workflow with clear status tracking and automated notifications.

    stateDiagram-v2
    [*] --> Draft
    Draft --> Submitted: Unit Head Submits
    Submitted --> UnderReview: Budget Team Reviews
    UnderReview --> Queried: Issues Found
    Queried --> Draft: Unit Head Revises
    UnderReview --> Approved: Budget Team Approves
    Approved --> Consolidated: All Units Approved
    Consolidated --> FinalReview: Management Reviews
    FinalReview --> Queried: Management Requests Changes
    FinalReview --> Published: Management Approves
    Published --> [*]

    6.1.2 Workflow Functional Requirements¶

    FR-WF-001: Draft Stage - Owner: Head of Business Unit / Operations Team - Actions: - Create budget entries - Edit budget entries - Attach justification documents - Save as draft (unlimited saves) - Run validation checks (soft validations only) - Status: "Draft - In Progress" - Notifications: None

    FR-WF-002: Submission - Trigger: User clicks "Submit for Review" - Pre-submission Validation: - All mandatory fields completed - All justifications meet minimum requirements - Required attachments uploaded - Hard validation rules passed - Actions: - Lock budget for editing by submitter - Change status to "Submitted - Pending Review" - Send notification to Budget Team - Log submission timestamp and user - Rollback: Submitter can recall submission before review starts (status returns to Draft)

    FR-WF-003: Budget Team Review - Owner: Budget Team - Actions: - View submitted budget - Compare to assumptions and guidelines - Add review comments - Request clarifications - Approve or Query - Status: "Under Review" - Notifications: - Budget Team notified on submission - Reminder if no action taken within 3 business days (configurable)

    FR-WF-004: Query and Revision - Trigger: Budget Team clicks "Query" and enters query comments - Actions: - Status changes to "Queried - Revision Required" - Budget unlocked for submitter to edit - Notification sent to Unit Head with query details - Query comments visible to submitter - Submitter Actions: - Revise budget entries - Respond to query comments - Resubmit for review - Status: Returns to "Submitted - Pending Review" upon resubmission - Audit Trail: All queries and responses logged with timestamps

    FR-WF-005: Budget Team Approval - Trigger: Budget Team clicks "Approve" - Actions: - Status changes to "Approved - Unit Level" - Budget locked from further editing - Notification sent to Unit Head confirming approval - Budget included in consolidation pool - Approval Metadata: - Approved by (user ID) - Approval timestamp - Approval comments (optional)

    FR-WF-006: Consolidation - Owner: Budget Team - Trigger: All required unit budgets approved - Actions: - System automatically consolidates all approved budgets - Generates consolidated financial statements - Calculates KPIs - Runs integration validation checks - Status changes to "Consolidated - Pending Management Review" - Notifications: Executive Management notified that consolidated budget is ready for review

    FR-WF-007: Management Final Review - Owner: Executive Management - Actions: - Review consolidated budget - Review financial statements - Review KPIs vs. targets - Run scenario analysis - Approve or request revisions - Status: "Final Review - Management" - Options: - Approve: Status changes to "Published - Final Budget" - Request Changes: Status changes to "Queried - Management Feedback" - Specific units can be sent back to Draft for revision - Or Budget Team can make adjustments at consolidated level

    FR-WF-008: Publication - Trigger: Management approval - Actions: - Status changes to "Published - Final Budget" - Budget data flows to GL as budget allocations - Budget becomes baseline for variance reporting - All users granted read-only access to final budget - Notification sent to all stakeholders - Lock: Published budget cannot be edited; amendments require formal budget revision process

    6.1.3 Workflow Configuration¶

    FR-WF-009: The system shall allow configuration of: - Approval hierarchy (single or multi-level) - Notification recipients and triggers - SLA timers and escalation rules - Mandatory vs. optional approval stages

    FR-WF-010: The system shall support parallel and sequential approval paths: - Parallel: Multiple units submit and get approved independently - Sequential: Certain budgets require approval before others can proceed (e.g., Operations budget before Business Unit budgets)

    7. Validation & Error Handling¶

    7.1 Validation Framework¶

    The system must implement a two-tier validation framework: Hard Validations (blocking) and Soft Validations (warnings).

    7.1.1 Hard Validations (Blocking)¶

    Hard validations prevent submission or progression to the next workflow stage. Users must resolve these errors before proceeding.

    FR-VAL-001: Mandatory Field Validation - Rule: All fields marked as "Required" must be populated - Trigger: On submission - Error Message: "The following required fields are empty: [Field Names]. Please complete all required fields before submitting." - Action: Prevent submission, highlight empty fields in red

    FR-VAL-002: Justification Minimum Length - Rule: Justification text must meet minimum character count (default: 50 characters) - Trigger: On submission - Error Message: "Justification for [GL Account] is too short. Minimum [X] characters required, you have entered [Y] characters." - Action: Prevent submission, highlight field

    FR-VAL-003: Placeholder Text Detection - Rule: Justification cannot contain placeholder text (e.g., "TBD", "To be determined", "N/A", "NA", "None") - Trigger: On submission - Error Message: "Justification for [GL Account] contains placeholder text. Please provide a meaningful justification." - Action: Prevent submission, highlight field

    FR-VAL-004: Numeric Range Validation - Rule: Numeric fields must be within acceptable ranges (e.g., interest rate between 0% and 100%, amounts > 0) - Trigger: On data entry and submission - Error Message: "[Field Name] value of [Value] is outside acceptable range of [Min] to [Max]." - Action: Prevent submission, highlight field

    FR-VAL-005: Date Logic Validation - Rule: Maturity date must be after drawdown date, budget year must be future year - Trigger: On data entry and submission - Error Message: "Maturity Date cannot be before Drawdown Date." - Action: Prevent submission, highlight fields

    FR-VAL-006: Required Attachment Validation - Rule: If budget line item exceeds threshold (e.g., $50,000), supporting document attachment is mandatory - Trigger: On submission - Error Message: "Budget amount for [GL Account] exceeds $[Threshold]. Supporting documentation is required." - Action: Prevent submission, highlight line item

    FR-VAL-007: Chart of Accounts Validation - Rule: All GL account codes must exist in active Chart of Accounts - Trigger: On data entry - Error Message: "GL Account Code [Code] is not valid or inactive." - Action: Prevent entry, provide lookup tool

    FR-VAL-008: Integration Consistency Validation - Rule: Balance Sheet must balance (Assets = Liabilities + Equity) - Trigger: On consolidation and publication - Error Message: "Balance Sheet does not balance. Assets: [Amount], Liabilities + Equity: [Amount], Difference: [Amount]. Please review budget entries." - Action: Prevent publication, provide drill-down to identify discrepancy

    FR-VAL-009: Cash Flow Reconciliation Validation - Rule: Cash Flow Statement ending cash must equal Balance Sheet cash - Trigger: On consolidation and publication - Error Message: "Cash Flow ending balance ([Amount]) does not match Balance Sheet cash ([Amount]). Difference: [Amount]." - Action: Prevent publication, provide reconciliation report

    7.1.2 Soft Validations (Warnings)¶

    Soft validations flag potential issues but allow users to proceed with justification or acknowledgment.

    FR-VAL-010: Budget Growth Threshold Warning - Rule: If budget line item exceeds prior year actual/budget by more than X% (configurable, e.g., 20%), flag for review - Trigger: On data entry and submission - Warning Message: "Budget for [GL Account] is [Y]% higher than prior year. Please confirm this is intentional and justified." - Action: Display warning banner, allow user to acknowledge and proceed, log acknowledgment in audit trail

    FR-VAL-011: Zero Budget Warning - Rule: If budget line item is zero but had non-zero value in prior year, flag for review - Trigger: On submission - Warning Message: "Budget for [GL Account] is zero, but prior year actual was [Amount]. Please confirm this is intentional." - Action: Display warning, require user acknowledgment

    FR-VAL-012: KPI Threshold Warning - Rule: If calculated KPI falls outside target range (e.g., Cost-to-Income Ratio > 60%, NPL Ratio > 5%), flag for management attention - Trigger: On consolidation - Warning Message: "[KPI Name] of [Value]% is outside target range of [Min]% to [Max]%. This may require management review." - Action: Display warning on KPI dashboard, send notification to Budget Team and Management

    FR-VAL-013: Liquidity Warning - Rule: If projected monthly cash flow shows negative balance, flag liquidity risk - Trigger: On consolidation - Warning Message: "Projected cash flow for [Month] shows negative balance of [Amount]. Funding gap identified." - Action: Highlight month in cash flow report, send notification to Budget Team

    FR-VAL-014: Regulatory Compliance Warning - Rule: If projected LDR, CAR, or other regulatory ratios approach or breach limits, flag for review - Trigger: On consolidation - Warning Message: "[Ratio Name] of [Value]% approaches/exceeds regulatory limit of [Limit]%. Immediate action may be required." - Action: Display prominent warning, send notification to Executive Management and Compliance

    FR-VAL-015: Incomplete Justification Warning - Rule: If justification meets minimum length but lacks specific keywords (e.g., "because", "due to", "based on"), flag as potentially generic - Trigger: On submission (optional, configurable) - Warning Message: "Justification for [GL Account] may be too generic. Consider providing more specific rationale." - Action: Display warning, allow user to revise or acknowledge

    7.2 Error Handling and User Guidance¶

    FR-VAL-016: The system shall provide user-friendly error messages that: - Clearly identify the problem - Specify the location (GL account, field name) - Suggest corrective action - Provide examples of acceptable entries where applicable

    FR-VAL-017: The system shall provide a "Validation Summary" panel showing: - Count of errors and warnings - List of all validation issues - Click-to-navigate to each issue - Ability to export validation report

    FR-VAL-018: The system shall support bulk validation with: - "Validate All" button to run all checks before submission - Progress indicator for long-running validations - Ability to fix errors and re-validate without full page reload

    FR-VAL-019: The system shall allow Budget Team to: - Configure validation rules (thresholds, mandatory fields, etc.) - Enable/disable specific soft validations - Create custom validation rules using formula builder

    11. Implementation Approach¶

    11.1 Phased Implementation¶

    Phase 1: Foundation (Months 1-3) - System setup and configuration - Chart of Accounts mapping - User role and permission configuration - Integration with GL (basic) - Assumptions Engine implementation

    Phase 2: Core Budget Module (Months 4-6) - Business Unit budget templates - ZBB logic implementation - Workflow and approval engine - Audit trail - Basic reporting

    Phase 3: Specialized Modules (Months 7-9) - Loan Portfolio Module - Borrowings and Funding Module - Automated financial statement generation - KPI engine

    Phase 4: Advanced Features (Months 10-12) - Scenario and sensitivity analysis - Advanced analytics and dashboards - Variance analysis (if actuals available) - Performance optimization - User training and change management

    Phase 5: Go-Live and Support (Month 12+) - Parallel run with existing process - User acceptance testing (UAT) - Go-live - Post-implementation support
